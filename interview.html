CHƯƠNG 01 — コア

Core Skills

Đây là những gì CV ghi bạn "thành thạo". Nhà tuyển dụng sẽ hỏi sâu, có thể yêu cầu code trực tiếp. Nắm chắc từng mục dưới đây.

言語の基礎JavaScript
Bị hỏi nhiều nhất

Closure

Một hàm "nhớ" được biến ở scope cha ngay cả sau khi hàm cha đã chạy xong.

function counter() {
  let count = 0;
  return () => ++count;
}
const inc = counter();
inc(); // 1
inc(); // 2 — vẫn nhớ "count"

Hoisting & Scope

  • var hoisted lên đầu function, khởi tạo undefined
  • let/const hoisted nhưng nằm trong "temporal dead zone" → truy cập trước khi khai báo sẽ lỗi
  • Block scope ({}) vs Function scope

"this" binding — hay bị hỏi bẫy

  • Trong function thường: this phụ thuộc vào cách gọi (object gọi nó)
  • Trong arrow function: this lấy theo scope cha (lexical this) — không tự có this riêng
  • call / apply / bind — dùng để ép this trỏ tới object chỉ định
const obj = {
  name: 'Huy',
  normal: function() { console.log(this.name); }, // 'Huy'
  arrow: () => { console.log(this.name); }        // undefined (this = window/global)
};

Event Loop

Trình tự chạy: Call Stack → Microtask Queue (Promise) → Macrotask Queue (setTimeout). Microtask luôn chạy trước macrotask kế tiếp.

console.log('1');
setTimeout(() => console.log('2'), 0);
Promise.resolve().then(() => console.log('3'));
console.log('4');
// Kết quả: 1, 4, 3, 2
Mẹo: Đây là câu hỏi kinh điển. Luôn giải thích được vì sao Promise chạy trước setTimeout dù cả hai đều "delay 0ms".

Prototype & Prototype Chain

Mọi object JS đều có link ngầm __proto__ trỏ tới object cha. class trong ES6 chỉ là syntactic sugar cho prototype. Khi truy cập 1 thuộc tính, JS tìm trên chính object trước, không có thì tìm ngược lên chuỗi prototype.

Array/Object methods hay dùng

  • map / filter / reduce — viết tay được reduce để tính tổng, group by
  • Spread/rest: {...obj}, [...arr]
  • Destructuring: const {name, age} = user
  • Promise.all vs Promise.allSettled: all fail ngay nếu 1 promise reject, allSettled luôn đợi hết và trả kết quả từng cái

Debounce / Throttle — thường bị bắt code tay

function debounce(fn, delay) {
  let timer;
  return (...args) => {
    clearTimeout(timer);
    timer = setTimeout(() => fn(...args), delay);
  };
}

function throttle(fn, limit) {
  let inThrottle;
  return (...args) => {
    if (!inThrottle) {
      fn(...args);
      inThrottle = true;
      setTimeout(() => inThrottle = false, limit);
    }
  };
}

Debounce: chỉ chạy sau khi ngừng gọi trong X ms (VD: search input). Throttle: giới hạn chạy tối đa 1 lần mỗi X ms (VD: scroll event, resize).

So sánh giá trị & sao chép

  • == so sánh có coerce type, === so sánh chặt cả type
  • Shallow copy ({...obj}) chỉ copy 1 cấp, nested object vẫn tham chiếu chung
  • Deep copy: structuredClone(obj) hoặc JSON.parse(JSON.stringify(obj)) (mất function, Date, undefined)

Kiểu dữ liệu & Type Coercion

  • Primitive: string, number, boolean, null, undefined, symbol, bigint
  • Reference: object, array, function — gán bằng tham chiếu, không phải copy giá trị
  • null vs undefined: undefined = chưa gán giá trị, null = cố tình gán "rỗng"
Q: Giải thích closure bằng ví dụ thực tế trong project của bạn
Gợi ý trả lời: liên hệ tới debounce trong ô search sản phẩm ở project Shoes Store/Phone Store — biến timer được "giữ" trong closure giữa các lần gọi hàm.
Q: let/const/var khác nhau ở điểm nào?
var: function-scope, hoisted với giá trị undefined, redeclare được. let: block-scope, không redeclare được cùng scope. const: block-scope, không gán lại được (nhưng nếu là object/array thì nội dung bên trong vẫn sửa được).
Q: Sự khác nhau giữa null và undefined?
typeof undefined === 'undefined', typeof null === 'object' (đây là 1 bug lịch sử của JS). Dùng == thì null == undefined là true, nhưng === thì false.
型システムTypeScript
Core

Interface vs Type

  • interface: mở rộng được bằng declaration merging, hợp cho object shape / props
  • type: linh hoạt hơn, dùng được cho union, intersection, primitive alias
type Status = 'idle' | 'loading' | 'error';
interface Product {
  id: string;
  price: number;
  status: Status;
}

Generic

function getFirst<T>(arr: T[]): T {
  return arr[0];
}

Generic giúp hàm dùng lại được với nhiều kiểu dữ liệu mà vẫn giữ type-safety.

Utility Types hay dùng

  • Partial<T> — mọi field thành optional (hữu ích khi update form)
  • Pick<T, K> — chọn ra vài field
  • Omit<T, K> — loại bỏ vài field
  • Record<K, V> — định nghĩa object dạng map
  • Readonly<T> — mọi field không thể gán lại

any vs unknown

any tắt hoàn toàn type-check (nguy hiểm). unknown an toàn hơn — bắt buộc phải kiểm tra kiểu trước khi dùng.

Type API response

Luôn định nghĩa interface cho response API để tránh lỗi runtime khi field bị thiếu/đổi tên.

Q: Vì sao dự án của bạn dùng TypeScript thay vì JavaScript thuần?
Bắt lỗi ngay lúc code thay vì runtime, tự động gợi ý (autocomplete) tốt hơn, dễ maintain khi project lớn/nhiều người làm, tài liệu hoá được kiểu dữ liệu qua interface.
構築ReactJS
Trọng tâm phỏng vấn

Virtual DOM & Reconciliation

React tạo cây Virtual DOM, so sánh (diff) với cây trước đó, chỉ cập nhật phần khác biệt lên DOM thật → tối ưu hiệu năng thay vì render lại toàn bộ.

Vì sao cần "key" khi render list

Key giúp React nhận diện phần tử nào thay đổi/thêm/xóa giữa các lần render, tránh re-render nhầm hoặc mất state. Không nên dùng index làm key nếu list có thể thay đổi thứ tự (thêm/xoá/sắp xếp lại).

useEffect — dependency array & cleanup

useEffect(() => {
  const controller = new AbortController();
  fetchData(controller.signal);
  return () => controller.abort(); // cleanup
}, [id]); // chạy lại khi "id" đổi
  • Mảng rỗng [] → chạy 1 lần khi mount
  • Không có mảng → chạy sau mỗi render (hiếm khi cần)
  • Cleanup function chạy trước lần effect kế tiếp hoặc khi unmount

useMemo / useCallback — khi nào thật sự cần

Chỉ dùng khi có tính toán nặng hoặc truyền callback xuống component con được bọc React.memo. Dùng tùy tiện sẽ làm code phức tạp mà không tăng hiệu năng đáng kể.

Custom Hook

function useDebounce(value, delay = 400) {
  const [debounced, setDebounced] = useState(value);
  useEffect(() => {
    const t = setTimeout(() => setDebounced(value), delay);
    return () => clearTimeout(t);
  }, [value, delay]);
  return debounced;
}

Nguyên nhân re-render & cách tối ưu

  • Parent re-render → mọi con re-render theo (trừ khi dùng memo)
  • State/props thay đổi (kể cả object mới nhưng giá trị giống nhau)
  • Giải pháp: tách component nhỏ, React.memo, tránh tạo object/function mới trong JSX mỗi render, dùng key hợp lý

Controlled vs Uncontrolled component

Controlled: giá trị input do React state quản lý (value + onChange). Uncontrolled: dùng ref để đọc giá trị trực tiếp từ DOM, ít re-render hơn nhưng khó validate real-time.

Lifting state up & Context API

Khi 2 component con cần chia sẻ state, đưa state lên component cha chung rồi truyền xuống qua props. Nếu phải truyền qua nhiều tầng ("prop drilling"), cân nhắc dùng Context API hoặc thư viện state management.

useRef dùng để làm gì

Lưu giá trị không gây re-render khi thay đổi, hoặc truy cập trực tiếp DOM node (VD: focus input, đo kích thước phần tử).

Q: React Fiber là gì?
Fiber là kiến trúc reconciliation từ React 16, cho phép chia nhỏ việc render thành các đơn vị công việc, ngắt/tiếp tục được → giúp UI không bị đơ khi render nặng (concurrent rendering).
Q: Khi nào bạn chọn dùng useReducer thay vì useState?
Khi state phức tạp, nhiều action thay đổi khác nhau, hoặc state kế tiếp phụ thuộc logic action cụ thể — useReducer giúp logic cập nhật state rõ ràng, tập trung một chỗ, giống mô hình Redux nhưng local trong component.
状態管理Redux Toolkit
vs TanStack Query
CV dùng cả 2 → chắc chắn bị hỏi
Câu hỏi chắc chắn gặp: "Vì sao project của bạn dùng cả Redux Toolkit lẫn TanStack Query? Không dùng 1 cái thôi được không?"

Phân biệt cốt lõi

  • Redux Toolkit → quản lý Client State: state thuộc về UI, không đến từ server (giỏ hàng, trạng thái modal, filter đang chọn, auth token trong bộ nhớ)
  • TanStack Query → quản lý Server State: dữ liệu lấy từ API (danh sách sản phẩm, đơn hàng) — tự động cache, refetch, xử lý loading/error, đồng bộ nhiều nơi dùng chung data

Vì sao không dùng Redux để fetch API luôn?

Nếu dùng Redux thuần để fetch API, bạn phải tự viết loading state, error state, cache, invalidate cache, retry... TanStack Query có sẵn tất cả, giảm rất nhiều boilerplate.

Cú pháp cơ bản

// Redux Toolkit
const cartSlice = createSlice({
  name: 'cart',
  initialState: [],
  reducers: {
    addItem: (state, action) => { state.push(action.payload); }
  }
});

// TanStack Query
const { data, isLoading, error } = useQuery({
  queryKey: ['products'],
  queryFn: fetchProducts
});

Redux Toolkit giải quyết vấn đề gì của Redux thuần?

Redux gốc cần viết action type, action creator, reducer riêng biệt, rất nhiều boilerplate. RTK gộp lại qua createSlice, tích hợp sẵn Immer nên viết state "mutate trực tiếp" mà thực chất vẫn immutable phía dưới.

Cache invalidation trong TanStack Query

Sau khi useMutation thành công (VD: thêm sản phẩm), gọi queryClient.invalidateQueries(['products']) để báo React Query fetch lại dữ liệu mới nhất.

Q: TanStack Query cache dữ liệu ra sao?
Mỗi query có queryKey duy nhất, dữ liệu được cache theo key đó. Khi component khác gọi cùng key, dữ liệu lấy từ cache ngay (stale-while-revalidate), sau đó tự động refetch ngầm nếu dữ liệu đã "stale".
Q: Nếu không dùng thư viện nào, bạn quản lý state global bằng cách nào?
Context API kết hợp useReducer cho state đơn giản/vừa. Với app lớn, quy mô nhiều, Redux Toolkit hoặc Zustand vẫn tốt hơn vì tránh re-render toàn bộ cây component khi Context thay đổi.

App Router vs Pages Router

App Router (thư mục app/) hỗ trợ Server Component mặc định, layout lồng nhau, streaming. Pages Router (pages/) là cách cũ, mọi thứ đều Client Component.

Server Component vs Client Component

Server Component render trên server, không gửi JS xuống client → nhẹ hơn. Client Component (khai báo 'use client') cần khi có state, event handler, hook.

Core
  • Nested route — route con render trong <Outlet /> của route cha
  • Dynamic route: /product/:id → đọc bằng useParams()
  • Điều hướng bằng code: useNavigate()

function ProtectedRoute({ children, role }) {
  const { user } = useAuth();
  if (!user) return <Navigate to="/login" />;
  if (role && user.role !== role) return <Navigate to="/403" />;
  return children;
}

Interceptor

Request interceptor thường dùng để gắn token tự động vào header. Response interceptor dùng để xử lý lỗi tập trung (VD: 401 → tự logout).

Loading / Error state

Luôn có 3 trạng thái khi gọi API: loading, success (data), error — giải thích được cách bạn xử lý cả 3 trong project.

Cancel request khi unmount

Dùng AbortController để hủy request khi component unmount, tránh warning "set state on unmounted component".

HTTP Status Code cần nhớ

200 OK · 201 Created · 400 Bad Request · 401 Unauthorized · 403 Forbidden · 404 Not Found · 500 Internal Server Error

GET/POST/PUT/PATCH/DELETE khác nhau ra sao

PUT thay toàn bộ resource, PATCH chỉ sửa 1 phần. Cả hai đều idempotent (gọi nhiều lần kết quả như nhau), còn POST thì không (gọi nhiều lần tạo nhiều bản ghi mới).

意匠HTML / CSS / Tailwind
Core

Box Model

Mỗi phần tử gồm: content → padding → border → margin. box-sizing: border-box giúp width/height tính luôn cả padding + border (thường dùng mặc định cho dễ tính toán layout).

Flexbox vs Grid

  • Flexbox: layout 1 chiều (hàng hoặc cột) — hợp cho navbar, danh sách item cùng hàng
  • Grid: layout 2 chiều (hàng và cột cùng lúc) — hợp cho layout tổng thể trang, dashboard

CSS Specificity

Thứ tự ưu tiên: inline style > ID > class/attribute/pseudo-class > element. !important ghi đè tất cả nhưng nên tránh dùng vì khó maintain.

Position

static (mặc định) · relative (dịch so với vị trí gốc) · absolute (so với cha gần nhất có position khác static) · fixed (so với viewport) · sticky (dính khi scroll qua ngưỡng).

Utility-first là gì (Tailwind)

Thay vì viết class CSS riêng, ghép các class tiện ích có sẵn (flex, p-4, text-lg) trực tiếp trong HTML/JSX. Ưu điểm: không phải đặt tên class, ít CSS thừa (purge unused). Nhược điểm: HTML dài, cần quen mắt.

Responsive / Mobile-first

Tailwind mặc định là mobile-first: viết style cho mobile trước, thêm breakpoint sm: md: lg: để override cho màn lớn hơn.

Dark Mode

Dùng class dark: kết hợp toggle class dark trên thẻ html, lưu preference vào localStorage.

Semantic HTML & Accessibility cơ bản

  • Dùng đúng thẻ: <button> cho hành động, không dùng <div onClick>
  • alt cho ảnh, label gắn với input
  • Đảm bảo điều hướng được bằng bàn phím (focus visible)

安全Bảo mật cơ bản
Mở rộng

XSS (Cross-Site Scripting)

Kẻ tấn công chèn script độc hại vào input rồi hiển thị lại cho user khác. React tự động escape giá trị render trong JSX nên khá an toàn, nhưng dangerouslySetInnerHTML thì không — cần sanitize trước khi dùng.

CSRF (Cross-Site Request Forgery)

Kẻ tấn công lừa user thực hiện hành động ngoài ý muốn trên site đã đăng nhập. Phòng chống bằng CSRF token, kiểm tra SameSite cookie.

CORS

Cơ chế trình duyệt chặn request cross-origin trừ khi server cho phép qua header Access-Control-Allow-Origin. Đây là lỗi rất hay gặp khi frontend và backend chạy khác domain/port lúc dev.

Lưu token an toàn

So sánh localStorage (dễ bị đánh cắp qua XSS) vs httpOnly cookie (an toàn hơn vì JS không đọc được, nhưng cần xử lý CSRF).

Vercel tự động build & deploy mỗi khi push code lên GitHub (CI/CD đơn giản), có Preview Deployment riêng cho mỗi Pull Request để review trước khi merge.

Environment Variables

Biến môi trường (API key, URL backend...) không hard-code trong source, quản lý qua file .env, biến bắt đầu bằng NEXT_PUBLIC_ (Next.js) hoặc VITE_ (Vite) mới lộ ra client.

CHƯƠNG 03 — 接触

Exposure Skills

CV ghi "Exposure to" — không phải thành thạo. Chỉ cần nắm khái niệm, demo được thứ cơ bản, không cần chuyên sâu.

実行環境Node.js
Exposure
  • Node.js = môi trường chạy JavaScript ngoài browser, dùng V8 engine
  • Khác browser: không có window/DOM, có hệ thống module (require/import), truy cập được file system
  • Non-blocking I/O — Node xử lý nhiều request cùng lúc mà không chặn luồng chính nhờ event loop
  • CommonJS (require) vs ES Module (import) — khác cách load module
  • npm/package.json — quản lý dependency, script chạy project
Q: Node.js single-thread nhưng vẫn xử lý được nhiều request cùng lúc, vì sao?
Node dùng 1 thread chính cho JS, nhưng các tác vụ I/O nặng (đọc file, gọi DB, network) được đẩy xuống libuv/thread pool ở background, xong thì callback được đưa lại vào event loop để xử lý tiếp — nên thread chính không bị block.
枠組みExpress.js
Exposure

Middleware

Hàm chạy giữa lúc nhận request và trả response: req → middleware → res. Dùng cho auth check, logging, parse body...

const express = require('express');
const app = express();
app.use(express.json());

app.get('/api/products', (req, res) => {
  res.json({ data: [] });
});

app.use((err, req, res, next) => {
  res.status(500).json({ message: err.message });
});

Nên nhớ luồng CRUD cơ bản: GET (đọc) · POST (tạo) · PUT/PATCH (sửa) · DELETE (xóa).

REST API convention

GET /products lấy danh sách, GET /products/:id lấy 1 item, POST /products tạo mới — đặt tên route theo danh từ số nhiều, dùng HTTP method thể hiện hành động thay vì nhét vào URL (VD: tránh /getProducts).

文書型MongoDB
Exposure
  • NoSQL, document-based — lưu dữ liệu dạng JSON-like (BSON), không cần schema cố định
  • Mongoose: thư viện định nghĩa Schema/Model để làm việc với MongoDB có cấu trúc hơn
  • Khi nào chọn MongoDB: dữ liệu thay đổi cấu trúc linh hoạt, cần scale nhanh, không cần join phức tạp
const productSchema = new mongoose.Schema({
  name: String,
  price: Number,
});
const Product = mongoose.model('Product', productSchema);
関係型SQL
Exposure
  • Câu lệnh cơ bản: SELECT ... FROM ... WHERE ..., JOIN, GROUP BY
  • Primary Key (khóa chính, duy nhất) vs Foreign Key (khóa ngoại, liên kết bảng khác)
  • SQL có schema cố định, quan hệ rõ ràng, hợp dữ liệu có cấu trúc chặt (đơn hàng, giao dịch). MongoDB linh hoạt hơn nhưng ít ràng buộc hơn.
容器Docker
Exposure
  • Container ≠ Virtual Machine: container chia sẻ chung OS kernel → nhẹ và khởi động nhanh hơn VM
  • Dockerfile — file định nghĩa cách build image
  • Image — bản đóng gói sẵn (code + môi trường), Container — instance đang chạy của image đó
  • Lý do dùng: đảm bảo môi trường chạy giống nhau ở mọi máy, tránh lỗi "chạy trên máy tôi thì được"
出版WordPress
Exposure

Chỉ cần nắm khái niệm: Theme (giao diện), Plugin (mở rộng chức năng), Custom Post Type (loại nội dung tùy chỉnh ngoài Post/Page mặc định). Hiếm khi hỏi sâu trừ khi công ty làm site WordPress.

CHƯƠNG 04 — 対話

Câu hỏi hành vi (Behavioral)

Không kém phần quan trọng so với câu hỏi kỹ thuật. Dùng công thức STAR: Situation – Task – Action – Result.

星型Công thức STAR
  • Situation — Bối cảnh: dự án gì, vấn đề gì xảy ra
  • Task — Nhiệm vụ: bạn cần làm gì
  • Action — Hành động: bạn đã làm cụ thể ra sao
  • Result — Kết quả: đạt được gì, học được gì
Gợi ý dùng project thật: Project Phone Store của bạn có phần "phân quyền admin/user + xử lý localStorage" — đây là chất liệu tốt để kể về 1 khó khăn kỹ thuật thực tế đã tự giải quyết.
質問集Bộ câu hỏi thường gặp
Q: Giới thiệu bản thân (Tell me about yourself)
Cấu trúc gợi ý: đang học năm cuối ngành gì, định hướng Frontend, kể ngắn 1-2 project nổi bật + tech stack, kết bằng lý do ứng tuyển vị trí này.
Q: Điểm mạnh / điểm yếu của bạn là gì?
Điểm mạnh nên gắn với dẫn chứng cụ thể (VD: tự học nhanh — dẫn chứng bằng việc tự làm HuyLearn để học JS/tiếng Anh). Điểm yếu nên chọn cái thật nhưng đã có hướng cải thiện, tránh nói yếu điểm liên quan trực tiếp tới công việc.
Q: Kể về 1 khó khăn kỹ thuật bạn từng gặp và cách giải quyết
Dùng STAR. Gợi ý: lỗi Tailwind v4 (@variant dark, bug @apply group) từng gặp ở PhoneHub — mô tả cách bạn debug, tìm tài liệu, và fix được.
Q: Bạn xử lý thế nào khi bất đồng ý kiến kỹ thuật với đồng đội?
Nhấn mạnh: lắng nghe lý do của người kia, đưa ra dẫn chứng/so sánh cụ thể (ưu nhược điểm), ưu tiên giải pháp tốt cho project chứ không phải "thắng-thua", nếu vẫn bất đồng thì để leader/team quyết định.
Q: Vì sao bạn chọn Frontend?
Có thể liên hệ sở thích thẩm mỹ tối giản Nhật Bản, thích nhìn thấy kết quả trực quan ngay khi code UI, quan tâm trải nghiệm người dùng.
Q: Bạn học công nghệ mới như thế nào?
Dẫn chứng thực tế: tự xây HuyLearn làm tài nguyên tự học, thói quen đọc doc chính thức, làm project nhỏ để thực hành ngay sau khi học lý thuyết.
Q: Tại sao chúng tôi nên chọn bạn cho vị trí Intern này?
Nhấn mạnh: đã có 3 project thực chiến gần với công việc thật (không chỉ học lý thuyết), khả năng tự học công nghệ mới nhanh, có nền tảng cả frontend lẫn hiểu biết cơ bản backend để phối hợp team tốt.
Q: Bạn có câu hỏi nào cho chúng tôi không? (luôn nên hỏi lại)
Gợi ý hỏi: "Tech stack chính team đang dùng là gì?", "Intern sẽ được mentor trực tiếp hay làm việc độc lập?", "Dự án đầu tiên em có thể tham gia là gì?" — thể hiện sự chủ động và quan tâm thật sự.

CHƯƠNG 06 — 戦略
